# Motion Guidelines

## When motion is allowed
- to explain hierarchy
- to confirm state changes
- to reinforce list-to-detail or planner-to-map movement

## When motion is not allowed
- when it delays a primary task
- when it distracts from route or timing comprehension
- when system reduce-motion is enabled

## Page transitions
- use short fades or directional slides
- avoid playful or elastic motion in core flows

## Skeletons
- use low-contrast placeholders for cards, itinerary rows, and panels
- keep loading states calm and readable on ivory backgrounds

## Route and map transitions
- motion may reinforce origin-to-destination continuity
- route readability must always win over visual flourish

## Reduce-motion considerations
- respect system settings
- replace complex transitions with fades or immediate state changes
